## Usage

```bash
mailman run [filepath] [options]
```

## Description

The `run` command executes an HTTP request and displays the response. It:

- Resolves all variables from environments, `.env` files, and command-line arguments
- Prompts for any unfilled variables at runtime
- Makes the HTTP request using the resolved configuration
- Displays response status, headers, and body
- Shows request duration

If no filepath is provided, you'll be prompted to select a request interactively from all available requests.

## Arguments

<ParamField path="filepath" type="string" optional>
  Path to the request file to run (relative to project root). If omitted, an interactive prompt will let you select from available requests.
</ParamField>

## Options

<ParamField path="--env" type="string" default="defaultEnvironment from config">
  Specify which environment to use for variable resolution. Short flag: `-e`
</ParamField>

<ParamField path="--var" type="string[]" optional>
  Set variables from the command line using key=value format. Can be used multiple times. Short flag: `-v`
</ParamField>

## Variable Resolution

Variables are resolved in the following order (later sources override earlier ones):

1. **Environment variables** - From `config.json` under the selected environment
2. **Default values** - From the `defaultValues` field in the request file
3. **`.env` files** - From `.env` (default) or `.env.{environment}` (e.g., `.env.prod`)
4. **Command-line variables** - From `--var` flags
5. **Runtime prompts** - For any remaining unfilled variables

### Example with Multiple Variable Sources

**config.json:**

```json
{
  "defaultEnvironment": "dev",
  "environments": {
    "dev": {
      "variables": {
        "baseUrl": "http://localhost:3000",
        "apiVersion": "v1"
      }
    }
  }
}
```

**.env:**

```
API_KEY=secret-key-from-env-file
```

**Request file** (`requests/get-user.json`):

```json
{
  "name": "Get User",
  "method": "GET",
  "url": "{{baseUrl}}/{{apiVersion}}/users/{{userId}}",
  "headers": {
    "Authorization": "Bearer {{API_KEY}}"
  },
  "defaultValues": {
    "userId": "1"
  }
}
```

**Command:**

```bash
mailman run requests/get-user.json --var userId=42
```

**Resolved request:**

- `baseUrl` → `http://localhost:3000` (from config.json)
- `apiVersion` → `v1` (from config.json)
- `userId` → `42` (from --var, overrides defaultValues)
- `API_KEY` → `secret-key-from-env-file` (from .env)

**Final URL:** `http://localhost:3000/v1/users/42`

## Interactive Variable Prompts

If any variables are not resolved from the sources above, you'll be prompted at runtime:

```bash
mailman run requests/get-user.json
```

**Prompt:**

```
? Enter value for userId: 123

Running: Get User
...
```

## Response Status Colors

The status code is colored for quick visual feedback:

- **Green** (2xx) - Success
- **Yellow** (3xx) - Redirection
- **Red** (4xx, 5xx) - Client/Server errors

## Cookie Support

If your request file includes a `cookies` field and your environment configuration defines cookies, they will be resolved and included in the request.

## Run a New Request (Ad-hoc)

You can create and run a request without saving it:

```bash
mailman run
```

Then select **"Run new request"** from the prompt and configure it interactively.

## Prerequisites

Before using `mailman run`, ensure you have:

1. Initialized a Mailman project with `mailman init`
2. A valid `config.json` file with environment configurations
3. At least one request file (or use interactive mode)

## Error Messages

### No config.json Found

```
No config.json found. Run 'mailman init' first.
```

**Solution:** Initialize your project with `mailman init`

### Connection Refused

```
Connection refused. Is the server running?
```

**Solution:** Ensure the target server is running and accessible at the specified URL.

### Request Failed

```
Request failed: getaddrinfo ENOTFOUND api.example.com
```

**Solution:** Check the URL and ensure it's correct. Verify network connectivity.

## Tips

- Use `mailman view` before `mailman run` to preview the resolved request
- Use `--var` to quickly override variables without editing files
- Response time is displayed for performance monitoring
- Use the interactive selection mode to quickly explore available requests
